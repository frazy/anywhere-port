# AnywherePort 项目结构与架构说明

本文档旨在为后续参与开发的工程师或 AI 助理提供整个工程的结构性视角，涵盖总体架构、目录结构、核心组件及其职责、通信约束以及其他项目级的代码规范约定。建议在开发新功能或大范围重构前仔细阅读本文档。

## 1. 总体架构模型 (Master / Agent)

AnywherePort 采用典型的 **Master-Agent（控制中心-执行节点）分布式架构**：

-   **Master 节点**：全局的管控中心。
    -   管理 `Web UI` 服务，为用户提供统一的操作面板；
    -   管理 `Auth`（鉴权中心），负责管理员登录、安全拦截；
    -   管理 `Hub`（集群核心），接受来自全球各处 Agent 节点的 WebSocket 连接，维护节点的上下线状态，即时向指定节点下发端口转发规则（增删改停）。
-   **Agent 节点**：纯粹的流量执行端。
    -   通过 WebSocket 长连接维持与 Master 的联系；
    -   根据 Master 下发的规则动态启停底层的 TCP/UDP 监听；
    -   (可选配置) 自动接管 Linux 系统上的 `ufw` 防火墙开放与关闭，保障无需通过 SSH 手动干预即可穿透；
    -   实现流量的高效代理转发，并周期性地将流量消耗、系统占用及**异常状态 (Error)** 上报给 Master。

## 2. 工程目录结构说明

为了保持项目的一致性和高内聚，目前代码采用扁平化的结构：

```text
anywhere-port/
├── cmd/
│   ├── master/main.go      # Master 节点主入口
│   └── agent/main.go       # Agent 节点主入口
├── pkg/
│   ├── auth/               # 认证模块（令牌校验、Web 鉴权拦截、防爆破）
│   ├── cluster/            # 集群通信模块（包含 Hub 服务端、Client 客户端以及消息契约）
│   ├── forward/            # 流量转发引擎（抽象 Engine、TCP/UDP 底层转发实现）
│   ├── limit/              # 流量限速模块（如 Token Bucket 限速器等）
│   └── web/                # Web UI 模块
│       ├── server.go       # Web 路由和接口实现
│       └── static/         # 前端静态资源，通过 embed.FS 打包至二进制文件
├── .agent/skills/          # 沉淀的专用开发技能脚本（如项目治理经验、自动化脚本等）
├── data/                   # 运行时数据存储目录（各节点配置与持久化统计）
├── dist/                   # 构建或自动生成的部署脚本输出目录
└── docs/                   # 官方说明文档集
```

## 3. 核心机制说明

### 3.1 集群通信协议 (WebSocket)
Master 与 Agent 的交互必须符合统一 JSON Msg 载荷约定。载荷基于 `type` 和 `payload` 进行强类型解析：
```json
{
  "type": "auth | heartbeat | sync | stats_report | cmd...",
  "payload": { ... }
}
```
**注意限制**：Agent 永远处于被动控制状态。任何转发规则的调整，必须由 Master `Broadcast` 到 Agent，Agent 更新内存规则后立即落地生效。

### 3.2 流量与存储闭环
流量持久化需要兼顾性能和可靠性。系统避免针对每一个字节读写磁盘：
1.  **Agent 侧**：每个 Agent 的 Engine 初始化时加载本地的 `data/agent_rules_{id}.json`，在转发中做原子流量累加。每隔 15 秒上报一次统计数据，并在内部定时 30s 持久化 `stats.json`。
2.  **Master 侧**：收到汇报数据后，同步至内存中的 `Hub`，更新缓存树，以便 Web 端拉取并即时展现。同时 Master 也会定时落地总览数据。

### 3.3 前端与渲染约定
*   **架构选型**：摒弃复杂的 Node.js/Vue/React 构建链路。系统以 **Go Template SSR (服务端渲染)** 为骨架，加上 **Vanilla JS (纯原生 JS)** 处理动态逻辑，辅以 Bootstrap 5 控制样式。
*   **组件开发**：针对重度弹窗、模态等，拆分到 `static/partials/*.tmpl`中。所有静态资源在 `web` 目录被 Go 原生 1.16+ 引入的 `go:embed` 技术自动打包入最终二进制文件中，以实现**零依赖单文件分发**。视觉交互务必遵循极致的紧凑设计和高度稳定的布局。
*   **交互容错**：由于是分布式异步架构，操作可能在 Agent 侧由于端口占用或 UFW 权限受到挫败。因此前端针对异常情况会有红色的高亮重叠呈现，并在操作列常备有 `/api/rules/{id}/retry` 机制方便用户一键重排重执行。

## 4. 建议与约束

-   所有数据操作（尤其存储路径），**严禁假定目录已存在**。程序中在落地前一律使用 `os.MkdirAll` 保证容错性。
-   如在代码层面实现了新的重构、重组或者发掘出了更好的设计范式，要求主动更新本项目的治理模型和 Skill （在 `.agent/skills/project_governance` 下更新说明）。
-   开发新功能时请留意向后兼容，例如更新 WebSocket 控制信令时。
